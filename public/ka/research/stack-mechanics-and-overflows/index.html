ბუფერის გადავსების მექანიკური გარდაუვალობა

November 28, 2025 ·4 წთ კითხვა ·Reverse Engineering Assembly Internals

„შავი ყუთის“ პრობლემა

დიდი ხნის განმავლობაში ასემბლერის ინსტრუქციებს ისე ვუყურებდი, როგორც შავ ყუთს. ვიცოდი, რომ CALL იძახებდა ფუნქციას, ხოლო RET ასრულებდა ფუნქციას. ვიცოდი ისიც, რომ თუ ბაფერში საკმარისი რაოდენობის “A”-ს ჩავტენი, პროგრამა დაიქრაშება.

თუმცა არ მესმოდა ამ “დარქაშვის” მექანიკა. ვერ ვიაზრებდი, რატომ აურევდა პროცესორი - სიზუსტისთვის შექმნილი მანქანა - ასე მარტივად მომხმარებლის მიერ შეყვანილ მონაცემებს კოდის მისამართებში.

ამ ჩემი გაუგებრობის აღმოსაფხვრელად საჭირო გახდა ცოტახნით კოდისთვის თვალის არიდება და უშუალოდ სტეკის მაჩვენებელზე (RSP) ფოკუსირება.

1. გარემო: სტეკი vs ჰიპი

პირველი ბარიერი მეხსიერების სწორი ვიზუალიზაცია იყო. ხშირად ვამბობთ ფრაზას „სტეკი და ჰიპი“, თუმცა ქცევის მხრივ ისინი აბსოლუტური საპირისპიროები არიან.

  • ჰიპი (ერთგვარი საწყობი): აქ მეხსიერების გამოყოფას ხელით ახორციელებ malloc ფუნქციით. იგი იზრდება ზემოთ, დაბალი მისამართებიდან მაღალი მისამართებისკენ. არის ქაოსური, მოქნილი და გრძელვადიანი შენახვისთვის.
  • სტეკი (ერთგვარი სამუშაო მაგიდა): აქ პროცესორი ასრულებს მიმდინარე, მყისიერ დავალებებს. ის არის ავტომატური, მკაცრად მოწესრიგებული და - რაც მთავარია - იზრდება ქვემოთ, მაღალი მისამართებიდან დაბალი მისამართებისკენ.

ეს “ქვემიმართულებით ზრდა” არის ამ ყველაფრის დედააზრი. როდესაც ფუნქციას ცვლადებისთვის ადგილი სჭირდება, ის მეხსიერებას კი არ უმატებს, არამედ სტეკის მაჩვენებელს აკლებს ახალი სამუშაო არეს გამოსაყოფად. როდესაც ეს ორი არეალი ერთმანეთს კვეთს, მეხსიერება ამოწურულად ითვლება.

2. RSP-ის მოძრაობა

სტეკის მაჩვენებელი (RSP) არ არის უბრალო კურსორი, ის არის საზღვარი.

  • ყველაფერი, რაც RSP-ს ზემოთაა მაღალ მისამართებზე, არის შენახული მონაცემები, რომლებიც უნდა შევინარჩუნოთ, მაგალითად ფუნქციიდან დაბრუნების მისამართები.
  • ყველაფერი, რაც RSP-ს ქვემოთაა დაბალ მისამართებზე, არის დროებითი სივრცე, ანუ ცარიელი მეხსიერება, რომელიც გამოყენებული არ არის ჯერ.

ინსტრუქციების შესრულებისას ჩვენ უბრალოდ ამ საზღვარს ვამოძრავებთ.

ალოკაცია არის გამოკლება

მაღალი დონის ენებში ჩვენ ცვლადებს ვაცხადებთ. ასემბლერში კი უბრალოდ ხაზი გადაგვაქვს.

  • SUB RSP, 0x20: ეს ქმნის 32-ბაიტიან ბაფერს. ჩვენ არაფერი შეგვიქმნია, უბრალოდ RSP 32 ბაიტით ქვემოთ ჩამოვწიეთ და განვსაზღვრეთ, რომ ეს სივრცე ახლა ჩვენია.
  • ADD RSP, 0x20: ეს ათავისუფლებს ბაფერს. ჩვენ RSP-ს ისევ ზემოთ ვაბრუნებთ. მონაცემები ტექნიკურად კვლავ იქ რჩება, თუმცა გადადის დროებით ზონაში და მზად არის შემდეგი ფუნქციის მიერ გადასაწერად.

3. კონფლიქტი: ჩაწერა vs ალოკაცია

ხარვეზის გასაგებად, შევხედოთ ამ მარტივ C პროგრამას:

int func() {
    char buffer[8]; // მოწყვლადობა აქ არის
    return 0xbeef;
}

int main() {
    func();
    return 0xfeeb;
}

როდესაც main იძახებს func ფუნქციას, ასემბლერის დონეზე სრულდება შემდეგი მექანიკური ნაბიჯები:

  1. call func - გამოძახება ორ რამეს აკეთებს. პირველ რიგში ასრულებს PUSH RIP ბრძანებას. ის იღებს მომდევნო ინსტრუქციის მისამართს და სვამს მას სტეკზე. ეს არის შენახული დაბრუნების მისამართი. შემდეგ კი ასრულებს JMP func-ს, რათა პროცესორი ფუნქციაზე გადაიყვანოს.
  2. sub rsp, 28h - ფუნქციის შიგნით, სტეკის მაჩვენებელი RSP მიდის ქვემოთ. ამ კონტექსტში გამოკლება ნიშნავს RSP-ის გადაადგილებას მეხსიერების უფრო დაბალ მისამართზე, რაც ჩვენი ბაფერისთვის და სხვა ოპერაციებისთვის სივრცეს ათავისუფლებს.
  3. ჩაწერის ოპერაცია - აქ ხდება მომხმარებლის მიერ მონაცემის შეყვანა, მაგალითად strcpy ან gets ფუნქციებით.
  4. RET - როდესაც ფუნქცია სრულდება, ის ახორციელებს RET ინსტრუქციას. ეს ბრძანება ასრულებს POP RIP ოპერაციას: უყურებს სტეკის თავს, იღებს იქ მყოფ მნიშვნელობას, რომელიც წესით შენახული დაბრუნების მისამართი უნდა იყოს, და სვამს მას ინსტრუქციების მაჩვენებელ RIP-ში.

მექანიკური ხარვეზი

ეს არის კონკრეტული მომენტი, რამაც ჩემთვის ყველაფერი ნათელი გახადა. აქ გვაქვს მიმართულებების ფუნდამენტური კონფლიქტი:

  1. სტეკი იზრდება ქვემოთ: სივრცეს გამოვყოფთ RSP-ის გადაადგილებით დაბალი მისამართებისკენ.
  2. ბაფერები იწერება ზემოთ: როდესაც მასივში მონაცემებს ვწერთ, ჩაწერა ხდება დასაწყისიდან, ანუ დაბალი მისამართიდან, ბოლოსკენ, ანუ მაღალი მისამართისკენ.

თუ ჩაწერის ოპერაციის დროს მეხსიერებას ვიზუალურად წარმოვიდგენთ, ეს შეჯახება აშკარა ხდება.

მეხსიერების განლაგება სტეკზე (მაღალი მისამართებიდან დაბალისკენ):

[ 0x00007fffffffe018 ]  <-- შენახული დაბრუნების მისამართი (ჩასმული CALL-ის მიერ)
[ 0x00007fffffffe010 ]  <-- ძველი ბაზისური მაჩვენებელი (Saved RBP)
[ 0x00007fffffffe000 ]  <-- 'buffer'-ის დასაწყისი (RSP აქ არის)

თუ ჩავწერთ 8 ბაიტს (“ABCDEFGH”), ბაფერს ზუსტად შევავსებთ 0x...008-მდე. მაგრამ თუ ჩავწერთ მონაცემს, რომელიც 16 ბაიტზე გრძელია:

Input: "AAAAAAAAAAAAAAAA" + "BBBBBBBB" + "DEADBEEF"

ჩაწერის ოპერაცია იწყება ბოლოდან და მიდის ზემოთ:

  1. “AAAAAAAAAAAAAAAA” ავსებს ბაფერს და პედინგს.
  2. “BBBBBBBB” გადაწერს ძველ ბაზისურ მაჩვენებელს.
  3. “DEADBEEF” გადაწერს შენახულ დაბრუნების მისამართს.

როდესაც RET შესრულდება, მან არ იცის, რომ სტეკი დაზიანებულია. ის უბრალოდ ამოიღებს 0xDEADBEEF მნიშვნელობას RIP მაჩვენებელში და პროგრამის შესრულება გადახტება ჩვენს მავნე მისამართზე: 0xDEADBEEF.

4. RET-ის ბრმა ნდობა

როდესაც ფუნქცია მუშაობას ასრულებს, ის უშვებს RET ინსტრუქციას. ეს არის მომენტი, სადაც ჩვენ ექსპლოიტს შეუძლია გააქტიურება.

RET არის მექანიკური ინსტრუქცია. ის აკეთებს ზუსტად ერთ რამეს:

  1. უყურებს ადგილს, სადაც RSP მიუთითებს.
  2. იღებს იქ მყოფ მნიშვნელობას.
  3. სვამს ამ მნიშვნელობას ინსტრუქციების მაჩვენებელში RIP.

ლოგიკური ხარვეზი მარტივია: RET არ ამოწმებს, თუ რა მნიშვნელობა ამოაქვს მას. ის უბრალოდ ვარაუდობს, რომ რადგან ეს მნიშვნელობა სტეკზე იდო ესეიგი ვალიდურია.

თუ ჩვენ მოვახერხეთ ამ სლოტის 0xDEADBEEF მნიშვნელობით გადაწერა, პროცესორი ვერ ხედავს უსარგებლო მონაცემებს. ის ხედავს მისამართს, რომელზეც შემდეგ უნდა გადავიდეს.

ის ბრმად გადადის 0xDEADBEEF მისამართზე.

შეჯამება

ბაფერის გადავსება არ არის მაგია ან ეზოთერული მეცნიერება. ეს არის ორი საპირისპირო ძალის მექანიკური შეჯახების შედეგი: სტეკი, რომელიც ადგილს ქვემოთ გამოყოფს და მონაცემები, რომლებიც ზემოთ იწერება.

პროგრამის შესრულების მიმდინარეობის ხელში ჩასაგდებად არ გვჭირდება კოდის ლოგიკის გატეხვა. ჩვენ უბრალოდ უნდა ჩავანაცვლოთ სტეკის თავი ჩვენი სასურველი მისამართით და დაველოდოთ, სანამ პროცესორი ბრმად შეასრულებს ინსტრუქციას.
