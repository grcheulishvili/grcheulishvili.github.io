პოლიმორფული ძრავის შემუშავება

December 8, 2025 ·5 წთ კითხვა ·MalDev Go Red Team

„Make“ ხრიკი და მისი ჩავარდნა

სტატიკური ანალიზისთვის გვერდის ავლის ჩემი პირველი მცდელობა იყო ძალიან მარტივი და დამწყების დონეზე. კომპილაციისას ბინარულ მეტამონაცემებში შემთხვევითი BUILD_ID სტრიქონის ჩასასმელად Makefile-ის მარტივი ხრიკი გამოვიყენე, რითაც ბინარული ფაილის ჰეში შეიცვლება.

# „Script Kiddie“ მიდგომა
LDFLAGS := -X main.buildID=$(shell openssl rand -hex 8)
go build -ldflags="$(LDFLAGS)" -o agent.exe # დაბილდვა ახალი მონაცემით რომ ჰეში შეიცვალოს

სპოილერი

ამ მიდგომამ არ იმუშავა. მიუხედავად იმისა, რომ ჰეში შეიცვალა, კოდის რეალური სტრუქტურა იდენტური დარჩა. .text სექციამ ზუსტად იგივე ენტროპია და ბაიტების თანმიმდევრობა შეინარჩუნა. შესაბამისად, Ghidra-ში კოდის შესრულების გრაფიკი (CFG) ზუსტად ერთნაირად გამოიყურებოდა. ეს ნიშნავს, რომ ფუნქციის ტანზე მორგებულმა ერთმა YARA წესმა ყველა „უნიკალური“ ბილდი მომენტალურად დაიჭირა.

ფაილისთვის სახელის გადარქმევა ან უაზრო მეტაინფორმაციის მიბმა არ არის პოლიმორფიზმი. საჭიროა თავად კოდის შეცვლა კომპილაციამდე.

კოდის მუტაცია

Regex და სტრიქონების ჩანაცვლების სკრიპტები ზედმეტად არამდგრადია კოდის მუტაციისთვის იმიტომ, რომ მარტივად შეიძლება სინტაქსი დაირღვეს. ამ დავალებს სწორად შესასრულებლად კოდი უნდა შემეცვალა ისე, როგორც ამას კომპილატორი ხედავს - სინტაქსური დატოტვის (AST) სახით.

ჩემი მიზანი იყო პრე-კომპილატორის შექმნა, რომელიც ბინარული ფაილის აწყობამდე შეცვლიდა Go-ს კოდის ლოგიკას. ამას დასჭირდა სამი კონკრეტული ნაბიჯი: სტრუქტურის შეცვლა, მონაცემების დამალვა და დეკრიპტაციის ლოგიკის ინექცია.

1. კომპილატორის წამკითხველის რევერსინგი

თავდაპირველად უნდა გამეგო, როგორ გამოიყურება Go კოდი მონაცემთა სტრუქტურის დონეზე. დავწერე სკრიპტი, რომელმაც ფუნქციის შიგნით არსებული მარტივი ტოლობის, _ = 1 + 2 AST სტრუქტურა ამოიღო.

2: *ast.AssignStmt {
   .  Lhs: [0: *ast.Ident { Name: "_" }]
   .  Tok: =
   .  Rhs: [0: *ast.BinaryExpr {
   .  .  .  X: { Kind: INT, Value: "1" }
   .  .  .  Op: +
   .  .  .  Y: { Kind: INT, Value: "2" }
   .  }]
}

ეს დამპი გახდა ჩემი რუქა. ამის საშუალებით უბრალოდ ტექსტის ცვლილება აღარ იყო ჩემი მიზანი, არამედ ობიექტების და და მათი სტრუქტურების ცვლილება, რომლებიც ჯამში კოდს წარმოადგენდნენ.

2. პირველი ვერსიის გაუმჯობესება: უსარგებლო კოდის ინექცია

ბინარული სიგნატურის შეცვლის პირველი ნაბიჯი ფუნქციების ბაიტების ოფსეტების არევაა. ამისათვის შევქმენი გენერატორი, რომელიც კოდში შემთხვევით არითმეტიკულ ოპერაციებს სვამს. ვიცი, რომ ქვემთ მოცემული ფუნქცია ამ დავალების უხეში რეალიზაციაა, თუმცა პროცესის დეტალურად გასააზრებლად მირჩევნია სტრუქტურა ხელით ავაწყო ვიდრე Go-ს AST პაკეტი დავიხმარო.

// აგენერირებს: _ = <RAND> <OP> <RAND> : მაგალითად _ = 1 + 3
func generateJunk() ast.Stmt {
    allowedTokens := []token.Token{token.ADD, token.SUB, token.MUL}

    lhs := []ast.Expr{
        &ast.Ident{ NamePos: token.NoPos, Name: "_" },
    }
    rhs := []ast.Expr{
        &ast.BinaryExpr{
            X: &ast.BasicLit{
                Kind:  token.INT,
                Value: strconv.Itoa(rand.IntN(1000)),
            },
            Op: allowedTokens[(rand.IntN(len(allowedTokens)))],
            Y: &ast.BasicLit{
                Kind:  token.INT,
                Value: strconv.Itoa(rand.IntN(1000)),
            },
        },
    }
    return &ast.AssignStmt{
        Lhs: lhs, 
        Tok: token.ASSIGN, 
        Rhs: rhs,
    }
}

ეს მათემატიკური ოპერაციები ფუნქციის ტანში პირდაპირ არ ჩამიყრია, რადგან კომპილატორმა შეიძლება ისინი მკვდარი კოდის სახით წაშალოს და ოპტიმიზაცია გაუკეთოს. ამის ნაცვლად, ეს დაგენერირებული კოდი მოვაქციე Opaque Predicate-ში - პირობით კონსტრუქციაში, როგორიცაა if 1==1, რომელიც ყოველთვის ჭეშმარიტია. ეს დეკომპილატორს აიძულებს შექმნას განტოტებული სტრუქტურა, რაც რევერსინგის ხელსაწყოებში ფუნქციის ვიზუალურ სიგნატურას მთლიანად ცვლის.

3. ჩაშებული სტრიქონის სიგნატურების განადგურება

უსარგებლო კოდი ცვლის სტრუქტურას, თუმცა ღია ტექსტური სტრიქონები, მაგალითად "cmd.exe", მაინც ჩანს strings ან YARA ხელსაწყოებისთვის. საჭიროა ყველა სტრიქონის პოვნა და მათი დაშიფვრა რომ ასეთ უსაფრთხოების ხელსაწყოებს გვერდი ავუაროთ.

კოდის AST-ის დასასკანერებლად golang.org/x/tools/go/ast/astutil პაკეტი გამოვიყენე. ჩემი პირველი მცდელობა წარუმატებელი აღმოჩნდა, რადგან განურჩევლად ვცადე ყველა ნაპოვნი სტრიქონის ჩანაცვლება. ამან გამოიწვია კომპილატორის შეცდომა, რადგან შემთხვევით მაგალითისთვის import "fmt" გადავაკეთე როგორც import xor("fmt"). Go-ს AST სტრუქტურაში იმპორტები მკაცრად განსაზღვრული ტიპებია და არა ცვლადი გამოსახულებები.

ეს პრობლემა მოვაგვარე ფილტრაციით, რათა უგულებელმეყო ImportSpec ნოდები და მუტაცია მხოლოდ გამოსახულებების კონტექსტში არსებულ სტრიქონებზე გავრცელებულიყო. ამის შემდეგ სტრიქონის ნოდი ჩავანაცვლე xor("encrypted_blob") ფუნქციის გამოძახებით.

astutil.Apply(node, func(c *astutil.Cursor) bool {
    // იმპორტების უგულებელყოფის ლოგიკა...
    
    // ჩანაცვლება xor ფუნქციით 
    newCall := &ast.CallExpr{
        Fun: ast.NewIdent("xor"),
        Args: []ast.Expr{
            &ast.BasicLit{ Value: fmt.Sprintf("%q", encrypted) },
        },
    }
    c.Replace(newCall)
    return true
}, nil)

4. პორტაბელურობა და ინექცია

სტრიქონები წარმატებით ჩავანაცვლეთ xor გამოძახებებით, თუმცა სამიზნე კოდს ეს ფუნქცია ჯერ არ გააჩნია. ბილდის გაშვების შემთხვევაში კომპილაცია ჩავარდებოდა. პრობლემა მდგომარეობს შემდეგში.

წარმოვიდგინოთ მაგალითი სადაც განვახორციელეთ კოდის ობფუსკაცია

func hello() string {
    // სტრიქონები ახლა დამალულია
    message := xor("\xc2\xcf\xc6...") 
}

მაგრამ ვიძახებთ xor() ფუნქციას რომელიც ამ კონკრეტულ ფაილში არ არის. როგორ უნდა დაინახოს კომპილატორმა ეგ ფუნქცია რომ გაშიფროს ტექსტი და runtime-ის დროს სწორად გამოიტანოს ინფორმაცია?

ამისთვის დავწერე ინჯექტორი, რომელიც კითხულობს xor დამხმარე ფუნქციის შაბლონს და მთლიან *ast.FuncDecl ობიექტს პროგრამულად ამატებს სამიზნე ფაილის AST-ის ბოლოში. ეს მალვეარს სრულად ავტონომიურს და თვითკმარს ხდის.

შედეგი

მუტატორი სატესტო აგენტზე სამჯერ გავუშვი სამი განსხვავებული ვარიანტის მისაღებად.

ქვემოთ ნაჩვენებია როგორ იკვრება მთლიანი სურათი: ძრავმა ჩასვა Opaque Predicates (ნაბიჯი 2), დაშიფრა სტრიქონები (ნაბიჯი 3) და დაამატა დამხმარე ფუნქცია (ნაბიჯი 4).

func hello() string {
    // ნაბიჯი 3: სტრიქონები ახლა დამალულია
    message := xor("\xc2\xcf\xc6...") 
    
    // ნაბიჯი 2: უსარგებლო კოდი ცვლის stack frame-ს
    if 1 == 1 {
        _ = 552 + 256
    } else {
        _ = 250 + 715
    }

    return message
}

// ნაბიჯი 4: ინექცირებული დამხმარე ფუნქცია
func xor(input string) string { ... }

ბილდის ფაილების შედარება აჩვენებს სამ მათემატიკურად განსხვავებულ ბინარულ ფაილს იდენტური ქცევით.

$ sha256sum agent_*
e84808...1af *agent_1.exe
79e3f2...759 *agent_2.exe
89edec...11e *agent_3.exe

შეჯამება

ამ ექსპერიმენტმა დაამტკიცა, რომ ნამდვილი პოლიმორფიზმი ბინარული ფაილის ბაიტების დონეზე სტრუქტურის შეცვლას მოითხოვს. თუ პროგრამის კოდს განვიხილავთ არა როგორც ტექსტს, არამედ როგორც მონაცემთა სტრუქტურას, შევძლებთ ყოველი დეპლოისთვის ფუნქციურად იდენტური, მაგრამ სტრუქტურულად უნიკალური ვარიანტების გენერირებას.

თუ გსურს თანამედროვე EDR სისტემების გვერდის ავლით მუშაობა, შეწყვიტეთ Makefile-ებში ან სტატიკური ცვლადების წერა და დაიწყეთ კომპილატორის Middleware ინსტრუმენტების შექმნა.

ზემოთ ნაჩვენები კოდი არის გამარტივებული კონცეფციის მტკიცებულება. ჩემი შიდა ძრავი ამ ლოგიკას აფართოებს დირექტორიების რეკურსიული დამუშავებისთვის, ფაილების კოლიზიების თავიდან ასაცილებლად და თითოეული ფაილისთვის შიფრაციის გასაღების როტაციისთვის. ეს ფუნქციები აქ გამოვტოვე კოდის მარტივად წაკითხვადობის შესანარჩუნებლად.
